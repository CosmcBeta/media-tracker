# Media Tracker — Project Plan

Disclaimer, this was generated by claude to assist me in building this.

A self-hosted, Rust-powered alternative to Sofa. Track movies, shows, albums, artists, books, games, and podcasts across custom lists — with full progress tracking, metadata from external APIs, and a mobile-friendly web UI.

---

## Vision

The goal is a personal media tracker you actually control. No subscription, no missing features, no compromises. You define the lists, the categories, and the tracking granularity. It lives on your mini PC at home and is accessible from any browser — including your phone.

## Data Model

### Core Tables

```sql
-- The main entity table for all media items
CREATE TABLE items (
    id          TEXT PRIMARY KEY,   -- UUID
    media_type  TEXT NOT NULL,      -- "movie", "show", "album", "artist", "book", "game", "podcast"
    title       TEXT NOT NULL,
    external_id TEXT,               -- ID from TMDB, Spotify, etc.
    metadata    TEXT,               -- JSON blob of fetched metadata
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);

-- User-defined lists (e.g. "Want to Watch", "Watching", "Finished")
CREATE TABLE lists (
    id         TEXT PRIMARY KEY,
    name       TEXT NOT NULL,
    icon       TEXT,
    created_at TEXT NOT NULL
);

-- Many-to-many: items can be in multiple lists
CREATE TABLE list_items (
    list_id    TEXT NOT NULL REFERENCES lists(id),
    item_id    TEXT NOT NULL REFERENCES items(id),
    added_at   TEXT NOT NULL,
    sort_order INTEGER DEFAULT 0,
    PRIMARY KEY (list_id, item_id)
);

-- Progress tracking (episode tracking for shows, page tracking for books, etc.)
CREATE TABLE progress (
    id         TEXT PRIMARY KEY,
    item_id    TEXT NOT NULL REFERENCES items(id),
    kind       TEXT NOT NULL,      -- "episode", "page", "percentage", "complete"
    value      TEXT,               -- e.g. "S02E04", "312", "75"
    note       TEXT,
    logged_at  TEXT NOT NULL
);
```

## API Design

All endpoints are under `/api/v1`.

### Items
```
GET    /items              — list all items (filterable by media_type)
POST   /items              — create new item (with optional metadata fetch)
GET    /items/:id          — get single item with full metadata
PATCH  /items/:id          — update item
DELETE /items/:id          — delete item

GET    /items/search?q=&type=   — search external APIs and return candidates
POST   /items/import            — import a search result as a local item
```

### Lists
```
GET    /lists              — get all lists
POST   /lists              — create a list
PATCH  /lists/:id          — rename / update list
DELETE /lists/:id          — delete list

GET    /lists/:id/items    — get all items in a list
POST   /lists/:id/items    — add item to list
DELETE /lists/:id/items/:item_id   — remove item from list
```

### Progress
```
GET    /items/:id/progress       — get progress history for an item
POST   /items/:id/progress       — log a new progress entry
DELETE /progress/:id             — delete a progress entry
```

## Rust Design Highlights

### Trait-Based Media Types

Define a `MediaItem` trait so each media type can implement its own metadata fetching, display formatting, and progress tracking logic:

```rust
pub trait MediaItem: Send + Sync {
    fn media_type(&self) -> &'static str;
    fn title(&self) -> &str;
    fn external_id(&self) -> Option<&str>;
    async fn fetch_metadata(&self, client: &ApiClient) -> Result<Metadata>;
    fn progress_kind(&self) -> ProgressKind;
}
```

This is a great place to explore `async fn` in traits (stable since Rust 1.75 via RPITIT).

### Layered Error Handling

Use `thiserror` to define a typed `AppError` enum that maps cleanly to HTTP responses:

```rust
#[derive(thiserror::Error, Debug)]
pub enum AppError {
    #[error("not found: {0}")]
    NotFound(String),
    #[error("external API error: {0}")]
    ExternalApi(#[from] reqwest::Error),
    #[error("database error: {0}")]
    Database(#[from] sqlx::Error),
    #[error("invalid input: {0}")]
    Validation(String),
}

impl IntoResponse for AppError { ... }
```

### Async API Client with Caching

The `ApiClient` struct wraps `reqwest::Client` and caches metadata in SQLite to avoid redundant external API calls. Fetches are concurrent using `tokio::join!` or `FuturesUnordered` when resolving multiple items at once.

### State with `Arc`

Axum's shared state will be an `Arc<AppState>` containing the database pool and API client — a natural introduction to `Arc` and interior mutability patterns in async Rust.

## Getting Started (Day One)

1. `cargo new media-tracker --workspace`
2. Add `backend` as a workspace member
3. Add dependencies to `backend/Cargo.toml`
4. Run `sqlx database create` and write your first migration
5. Stand up a single `GET /health` endpoint in Axum that returns `200 OK`
6. Write a single `POST /items` and `GET /items` handler with in-memory storage first, then swap to SQLite

Resist the urge to build everything at once. Phase 1 done well is a better foundation than all 6 phases half-finished.

# Backup Project Ideas

## Backup 1 — Terminal Music Player

A TUI music player for local audio files with a library browser, queue management, and optional last.fm scrobbling or lyrics fetching.

**Core crates:** `ratatui` (TUI), `rodio` (audio playback), `lofty` (audio metadata/tags), `tokio` (async API calls), `reqwest` (lyrics/scrobbling), `sqlx` + SQLite (library DB)

**Rust depth:** Async event loop balancing TUI rendering with audio playback, trait abstractions for audio backends, lifetime challenges around borrowed audio buffers, layered error handling across IO + audio + network.

**Why it's good:** Directly useful if you ever switch to local music, touches a wide spread of Rust crates, and the TUI looks great in a demo.

**Caveat:** You currently only stream music, so day-to-day utility is limited unless you add a streaming backend (e.g. wrapping the Spotify API), which adds significant scope.

## Backup 3 — 3D Printer Chess Robot (Arduino Mega 2560)

Repurpose a broken 3D printer (with a non-functional heating element) as a robotic chess board. A chess engine (e.g. Stockfish via UCI protocol) sends moves, which are translated into G-code and sent to the printer's motion system to physically move pieces on a board.

**Stack:** Rust binary on a host PC (Mac or mini PC) communicating with the Arduino over serial (`serialport` crate), UCI protocol integration with Stockfish, move-to-G-code translation layer, optionally a small TUI to watch the game live.

**Core crates:** `serialport` (serial communication), `tokio` (async coordination), `chess` or custom move representation, `ratatui` (optional game viewer)

**Rust depth:** Async serial I/O, state machine design for the robot's movement queue (can't send the next move until the printer finishes the current one), error handling across hardware failures, UCI protocol parsing.

**Why it's good:** Extremely unique and memorable — nobody else at an internship interview has this. Combines software, hardware, and a fun demo you can literally show people. The hardest part (heating element) is already out of scope.

**Caveats:** Hardware in the loop makes debugging slow and frustrating. Requires physical access to the printer. The project was already started and stalled once, so re-motivating might take effort. Best framed as a fun side project alongside a more polished resume piece rather than the main event.
